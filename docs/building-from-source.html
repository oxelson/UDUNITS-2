    Building from Source

    UDUNITS-2 Documentation Building from Source

    This page describes how to build and install UDUNITS-2 from source code for C and C-based applications.

    When To Build From Source

    Building from source is recommended if you need:

    • A specific UDUNITS-2 version not available via your package manager
    • A custom installation prefix (e.g., /opt, $HOME/local)
    • A controlled build environment (HPC, containers, CI)
    • Windows or nonstandard platforms

    For most users, installing via a system package manager is simpler. See the package-manager installation guide if that applies to you.

    Supported Platforms

    UDUNITS-2 can be built from source on:

    Platform Status Notes
    Linux (POSIX) ✅ Fully supported Most common build target
    macOS ✅ Fully supported Uses system toolchain
    Windows ⚠️ Supported Requires POSIX-like environment or CMake toolchain

    Build Dependencies

    Dependency / Tool Required Purpose
    C compiler (C99+) Yes Builds the UDUNITS-2 C library
    Expat XML parser Yes Parses the udunits2.xml units database
    Make Yes Orchestrates compilation and installation
    Texinfo (makeinfo) Yes* Generates GNU info documentation when building from Git
    GNU Autotools (autoconf, automake, libtool, m4) Conditional Required when generating the configure script from a Git checkout
    pkg-config No Helps downstream projects discover installed UDUNITS-2
    Note: When building from a Git repository clone, the configure script must be generated using GNU Autotools. Release archives include pre-generated build scripts and do not require Autotools. When building from a Git checkout, Texinfo is required because the build system regenerates the GNU info manual from udunits2lib.texi. Release distributions include pre-generated documentation and typically do not require Texinfo.

    Obtaining The Source Code

    Obtain the latest version of UDUNITS-2 from GitHub:

    git clone https://github.com/Unidata/UDUNITS-2.git
cd UDUNITS-2

    Default Build (Most Users)

    Use a standard build when you are on a Unix-like system and you are OK with both the UDUNITS-2 and system defaults:

    Default Details
    Installation Location /usr/local
    Libraries Shared and static (system default behavior)
    Toolchain System compiler and linker

    Run the following commands in order to build UDUNITS-2 in the default location (/usr/local):

    autoreconf -i     # requires GNU autotools 

./configure
make
make install

    Custom Builds

    Custom build configurations for UDUNITS-2 are typically needed when installing on shared systems, HPC clusters, or CI environments where default system paths, compilers, or dependencies differ from standard layouts. They allow you to control installation locations, toolchains, and dependency discovery to ensure compatibility, isolation, and reproducible builds.

    More Information: For a full list of supported options, see the Configure Script Options section or run ./configure --help directly from the source directory.

    Custom Installation Prefix

    Specify the --prefix=DIR option during configuration to specify a custom location for the UDUNIT-2 installation. (This is common on shared systems or when avoiding conflicts with package-managed installs.)

    For example, the following will install UDUNITS-2 in /opt/udunits:

    ./configure --prefix=/opt/udunits
make
make install

    Nonstandard Dependency Locations

    If a dependency is installed in a nonstandard location, specify its paths explicitly using the following options:

    Option Purpose
    CPPFLAGS Locate nonstandard header paths (e.g. Expat)
    LDFLAGS Locate nonstandard library paths

    For example, if Expat is installed in a nonstandard location, specify its paths explicitly:

    ./configure \
  CPPFLAGS="-I/path/to/expat/include" \
  LDFLAGS="-L/path/to/expat/lib"

    Windows Installations

    Building UDUNITS-2 on Windows typically requires:

    • MSYS2 or MinGW environment
    • CMake-based or autotools-compatible toolchain
    • Manual handling of library paths

    Windows builds are best attempted only when package-managed or Conda-based solutions are not viable.

    HPC / Shared System Builds

    When building UDUNITS-2 on a cluster or other shared system, system-wide installation is often not allowed and the environment is typically more customized than a desktop system. To avoid conflicts with system-provided software, it is recommended to install UDUNITS-2 into a user- or site-specific location.

    On these systems, shared libraries are commonly used, compilers are often provided by the site, and dependencies such as Expat may live in nonstandard locations. As a result, you will usually need to guide the build system by specifying an installation prefix and, in some cases, explicit include and library paths.

    The following options are commonly used in HPC environments:

    Option / Variable Purpose
    --prefix=DIR Install UDUNITS-2 into an isolated location
    CPPFLAGS Specify nonstandard header paths (e.g. for Expat)
    LDFLAGS Specify nonstandard library paths

    For example:

    ./configure \
  --prefix=/path/to/UDUNITS-2 \
  CPPFLAGS="-I/path/to/expat/include" \
  LDFLAGS="-L/path/to/expat/lib"
make
make install

    CI / Automated Builds

    When building UDUNITS-2 in continuous integration (CI) pipelines or other automated, reproducible environments, the emphasis is usually on speed, consistency, and clear build output. CI builds are typically performed in a temporary workspace, use shared libraries only, and avoid unnecessary build steps or artifacts.

    To keep builds fast and logs easy to read, it is common to disable dependency tracking, reduce build verbosity, and skip static library generation. These settings are well suited to one-time, non-interactive builds where the software is installed, tested, and discarded.

    The following options are commonly used in CI environments:

    Option Purpose
    --disable-dependency-tracking Speeds up one-time builds
    --enable-silent-rules Produces cleaner, less verbose CI logs
    --disable-static Reduces build artifacts and build time

    For example:

    ./configure \
  --prefix=$PWD/install \
  --disable-dependency-tracking \
  --enable-silent-rules \
  --disable-static
make
make install

    Debugging / Development Builds

    When developing or debugging with UDUNITS-2 itself, the focus is on visibility rather than build speed. In this mode, builds are typically configured to produce binaries with debug symbols and, when needed, additional instrumentation such as code coverage.

    These builds are often installed into a non-system prefix to avoid interfering with production installations and may include either shared or static libraries. Verbose build output is usually preferred to make it easier to diagnose build and linking issues.

    The following options are commonly used for debug and development builds:

    Option Purpose
    --enable-debug Enable debugging symbols
    --enable-coverage Enable code coverage instrumentation
    --disable-silent-rules Produce verbose build output

    For example:

    ./configure \
  --enable-debug \
  --disable-silent-rules
make
    Warning: Debug and coverage configuration options are intended for development and testing only and should not be used in production environments, as they can significantly increase binary size, reduce performance, and introduce unnecessary instrumentation that is inappropriate for security-sensitive deployments.

    Files Installed

    After building and installing UDUNITS-2 from source, the installation will typically place binaries, libraries, headers, and supporting data under the chosen installation prefix.

    The exact paths may vary depending on the --prefix used during configuration.

    If no prefix is provided, the default location is /usr/local.

    Component Typical Location Purpose
    udunits2 PREFIX/bin/udunits2 Command-line unit conversion tool
    libudunits2 PREFIX/lib/libudunits2.so (Linux)
    PREFIX/lib/libudunits2.dylib (macOS)    		 Shared library used by applications
    udunits2.h PREFIX/include/udunits2.h C header for developing software using UDUNITS-2
    udunits2.xml PREFIX/share/udunits/udunits2.xml XML database defining units and conversions
    Documentation PREFIX/share/doc/udunits Installed documentation files

    Environment Variables

    When installing from source, you may need to adjust environment variables depending on your prefix.

    Variable When Needed
    PATH If udunits2 is not in a standard bin directory
    LD_LIBRARY_PATH If shared libraries are outside system paths
    UDUNITS2_XML_PATH Only if XML database is moved or customized

    Example:

    export PATH=/opt/udunits/bin:$PATH
export LD_LIBRARY_PATH=/opt/udunits/lib:$LD_LIBRARY_PATH
    Warning: Avoid setting LD_LIBRARY_PATH globally on systems that also use package-managed libraries.

    Verifying The Build

    After installing UDUNITS-2, verify that the installation is working before building dependent software.

    A correct installation requires the following components to be discoverable:

    • the udunits2 executable
    • the development headers (udunits2.h)
    • the UDUNITS-2 library (libudunits2)
    • the units database (udunits2.xml)

    The Installation Health Check guide provides commands and scripts that confirm these components are installed and accessible. It can quickly detect common issues such as missing headers, missing libraries, incorrect include or library paths, or a misconfigured units database.

    Troubleshooting

    When building UDUNITS-2 from source, installation issues typically occur during one of the following stages:

    Dependencies → Build Configuration → Compilation and Linking → Runtime → Environment Conflicts → Platform-Specific Issues

    Before using the troubleshooting tables below, run the checks in the UDUNITS-2 Installation Health Check to verify that the UDUNITS-2 executable, headers, libraries, and units database are correctly installed.

    Dependency Issues

    These errors occur when required libraries cannot be found during the configuration stage.

    Symptom Cause Diagnosis Fix
    configure: cannot find EXPAT function XML_StopParser Expat library missing or not detected Inspect config.log Install the Expat development package
    expat.h: No such file or directory Expat headers not installed Search include directories Install Expat development headers
    ld: cannot find -lexpat Linker cannot locate Expat library Check library paths Install Expat library or add -L/path/to/expat/lib

    Build Configuration Issues

    These problems occur during the configure or CMake stage when paths or environment variables are not set correctly.

    Symptom Cause Diagnosis Fix
    Configure fails to detect dependencies Environment variables not configured correctly Inspect config.log Set CPPFLAGS and LDFLAGS appropriately
    udunits2.h: No such file or directory Headers installed in non-standard location Locate installed headers Add -I/path/to/include

    Compilation And Linking Issues

    These errors occur when compiling or linking applications against UDUNITS.

    Symptom Cause Diagnosis Fix
    ld: cannot find -ludunits2 Library path not visible to linker Inspect library install location Add -L/path/to/lib
    undefined reference to ut_* functions Program not linking against UDUNITS-2 Inspect compile command Add -ludunits2

    Runtime Issues

    These problems appear after successful compilation when the program is executed.

    Symptom Cause Diagnosis Fix
    Runtime error loading libudunits2.so Library path not registered with dynamic linker Run ldd program Run ldconfig or update LD_LIBRARY_PATH
    Couldn't open units database XML units database not found Search installation directory for udunits2.xml Set UDUNITS2_XML_PATH

    Environment and Installation Conflicts

    These issues occur when multiple versions of UDUNITS-2 exist on the system.

    Symptom Cause Diagnosis Fix
    Program links against wrong UDUNITS-2 version Multiple installations present Inspect linked libraries with ldd Remove duplicate installations or adjust include/library paths

    Windows-Specific Issues

    These problems occur when building UDUNITS-2 using Visual Studio or Windows toolchains.

    Symptom Cause Diagnosis Fix
    udunits2.h not found in Visual Studio Include path not configured Check project include directories Add UDUNITS-2 include directory
    udunits2.lib not found during linking Import library not configured Inspect linker settings Add library directory and dependency
    libudunits2.dll not found at runtime DLL not in system search path Run program from command prompt Add directory to PATH or copy DLL next to executable

    Best Practices

    • Use a custom prefix when installing alongside system package manager versions to avoid conflicts
    • Verify the build immediately after installation
    • Avoid mixing source-built and system-installed libraries

    Using UDUNITS-2 in C Projects

    After installation, the UDUNITS-2 header files and libraries are typically placed in standard directories beneath the chosen installation prefix. The exact locations depend on the prefix used during installation (for example /usr, /usr/local, or a custom directory such as /opt/udunits).

    In most cases the files will be organized as follows:

    • Headers: include/udunits2.h
    • Libraries: lib/libudunits2.so (Linux) or lib/libudunits2.dylib (macOS)

    The header file provides the function declarations and data structures required to compile applications that use the UDUNITS-2 C API. The shared library contains the compiled implementation that your program links against at build time and loads at runtime.

    When compiling a C program that uses UDUNITS-2, the compiler must be able to locate both the header files and the library. This is usually done by specifying the include and library paths with the -I and -L compiler flags, and linking the UDUNITS-2 library using -ludunits2.

    Example Compile Command

    gcc example.c -I/opt/udunits/include -L/opt/udunits/lib -ludunits2

    In this example:

    • -I/opt/udunits/include tells the compiler where to find udunits2.h
    • -L/opt/udunits/lib tells the linker where to find the UDUNITS-2 library
    • -ludunits2 links the program against libudunits2

    Adjust these paths as needed to match the installation prefix used on your system.

    NetCDF + UDUNITS-2 Integration

    How NetCDF Uses UDUNITS-2

    UDUNITS-2 provides unit parsing and conversion functionality used by:

    • NetCDF command-line tools
    • NetCDF-based libraries and applications
    • Higher-level bindings such as Python, R, and Fortran

    NetCDF expects:

    • libudunits2 available at runtime
    • Access to the udunits2.xml units database

    Recommended Build Order

    When building the Python, R, or NetCDF software stack from source, it is generally recommended to follow this dependency order:

    Expat → UDUNITS-2 → NetCDF-C → NetCDF-Fortran

Expat → UDUNITS-2 → NetCDF-C → NetCDF-C++

Expat → UDUNITS-2 → NetCDF-C → Python

Expat → UDUNITS-2 → NetCDF-C → R

    All components should be installed using the same installation prefix (for example, /usr/local, /opt/netcdf, or another consistent directory). Using a shared prefix keeps header files, libraries, and configuration scripts in predictable locations, which simplifies compiler and linker configuration during the build process.

    Maintaining a consistent prefix also reduces the risk of mixing system-installed libraries with custom-built versions. This helps avoid subtle runtime or linking issues that can occur when multiple versions of the same dependency exist on the system.
