    Troubleshooting API Issues

    UDUNITS-2 Documentation Troubleshooting API Issues

    Most UDUNITS-2 C API usage issues fall into a small number of common categories.

    Before using the troubleshooting tables below, ensure that the unit system context, unit definitions, and conversion workflow follow the expected lifecycle described in the UDUNITS-2 C API reference.

    Unit Parsing Issues

    These errors occur when converting text unit specifications into ut-unit objects using ut-parse().

    Symptom Cause Diagnosis Fix
    ut_parse() returns NULL Unit string not recognized by grammar Enable error handler using ut_set_error_message_handler() and inspect message Verify spelling and grammar using udunits2 -H
    Valid-looking unit fails to parse using ut_parse() String uses unsupported syntax Compare with canonical examples Rewrite using UDUNITS grammar conventions
    Prefix interpreted incorrectly Ambiguous prefix usage Inspect parsed output using ut_format() Use explicit unit symbols
    Expression parses but produces unexpected unit Operator precedence misunderstood Review parentheses in unit expression Add explicit grouping
    Parsing fails for exponent notation in ut_parse() Unsupported superscript characters used Check character encoding Use ASCII exponent syntax (m^2)

    Unit-System Consistency Issues

    These issues occur when units belong to different ut_system* contexts.

    Symptom Cause Diagnosis Fix
    ut_are_convertible() returns 0 unexpectedly Units created from different ut_system instances Verify system pointer used when calling ut_parse() Ensure all units originate from the same ut_system
    Conversion fails between equivalent units Units defined in separate XML databases Compare system objects created by ut_read_xml() Load units from the same XML database file
    Custom unit cannot convert with standard unit Unit added to different system instance Trace system creation calls using ut_new_base_unit() Define custom units within the same ut_system context

    Conversion Semantics Issues

    These problems occur when computing numeric conversions using ut_get_converter().

    Symptom Cause Diagnosis Fix
    Conversion produces incorrect numeric results Units not dimensionally compatible Verify compatibility using ut_are_convertible() before calling ut_get_converter() Confirm dimensional equivalence of both ut_unit objects
    Temperature conversion produces unexpected offset Affine conversion required for temperature scales Inspect converter created by ut_get_converter() and evaluate results using cv_convert_double() Ensure correct temperature units (e.g., absolute vs relative) are used
    Converter creation fails Units not compatible Check compatibility using ut_are_convertible() Convert through a compatible intermediary unit or revise unit definitions
    Conversion works but values appear scaled incorrectly Prefix or exponent misinterpreted during parsing Inspect parsed representation using ut_format() after ut_parse() Validate input unit string and exponent syntax

    Memory Ownership Issues

    These issues occur when managing object lifetimes for ut_unit and converter objects returned by ut_get_converter().

    Symptom Cause Diagnosis Fix
    Memory leak detected Unit objects not freed Audit lifecycle of objects created by ut_parse(), ut_scale(), or ut_multiply() Release units using ut_free()
    Crash during cleanup Object freed multiple times Trace ownership of objects returned by ut_clone() or ut_parse() Ensure each ut_unit object is freed exactly once using ut_free()
    Converter object persists unexpectedly Converter not explicitly freed Inspect lifecycle of converter created by ut_get_converter() Free converter using cv_free()
    Application frees memory incorrectly Using free() instead of API lifecycle functions Search code for incorrect deallocation of ut_unit or converter objects Use ut_free() and cv_free() instead of free()

    Error Handling Issues

    These issues occur when diagnostic messages are not captured or interpreted correctly.

    Symptom Cause Diagnosis Fix
    Function returns NULL with no explanation Error handler not configured Register custom handler using ut_set_error_message_handler() Configure error reporting before calling functions such as ut_parse()
    Unexpected error messages Handler state persists across calls Inspect global error handler configuration set by ut_set_error_message_handler() Reset or replace handler prior to subsequent API calls
    Error messages missing context Handler does not include sufficient diagnostic detail Enable structured logging within custom handler registered via ut_set_error_message_handler() Add contextual logging around calls to ut_parse() or ut_get_converter()

    Unit Definition Issues

    These issues arise when working with built-in or custom unit definitions.

    Symptom Cause Diagnosis Fix
    Expected unit symbol not recognized Symbol not defined in unit database Inspect definitions loaded via ut_read_xml() Use alternative symbol or define custom unit using ut_new_base_unit()
    Unit interpreted differently than expected Symbol conflict or ambiguity Inspect parsed structure using ut_parse() and format using ut_format() Use explicit unit expression (e.g., products via ut_multiply())
    Alias from UDUNITS-1 not recognized Definition removed or renamed in UDUNITS-2 database Compare legacy symbols with units loaded via ut_read_xml() Use updated canonical symbol recognized by ut_parse()
    Dimensionless unit behaves unexpectedly Unit defined incorrectly or missing dimensionless representation Inspect unit structure returned by ut_parse() or constructed using ut_scale() Define explicitly dimensionless unit using ut_scale() or equivalent identity unit

    Encoding Issues

    These problems occur when unit strings contain unsupported character encodings.

    Symptom Cause Diagnosis Fix
    Parsing fails for valid-looking symbols UTF-8 characters not recognized by parser Inspect encoding flags used with ut_parse() Use ASCII equivalents when calling ut_parse()
    Superscript characters fail to parse Non-ASCII exponent notation used Examine byte representation of input passed to ut_parse() Replace superscripts with ASCII exponent syntax (^) before calling ut_parse()
    Unit string behaves differently across environments Locale differences affect parsing behavior Compare locale and encoding assumptions when invoking ut_parse() Normalize input encoding prior to parsing

    Legacy Migration Issues

    These issues occur when porting legacy application from UDUNITS-1 to UDUNITS-2.

    Symptom Cause Diagnosis Fix
    Previously valid unit no longer recognized Database definitions updated in UDUNITS-2 Compare definitions loaded via ut_read_xml() Update unit expressions passed to ut_parse()
    Legacy API assumptions fail Function behavior changed between UDUNITS-1 and UDUNITS-2 Review semantics of ut_parse(), ut_get_converter(), and ut_are_convertible() Refactor code to follow UDUNITS-2 unit-system lifecycle
    Conversion results differ from UDUNITS-1 Dimensional analysis rules more strictly enforced Compare dimensional compatibility using ut_are_convertible() Update conversion workflow using ut_get_converter()
    Symbol interpreted differently Legacy alias removed or renamed in XML database Inspect definitions loaded via ut_read_xml() Use canonical symbol recognized by ut_parse()
