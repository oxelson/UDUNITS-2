    Database File Structure

    UDUNITS-2 Documentation Database File Structure

    The UDUNITS-2 unit database is implemented as a collection of XML documents that together define the vocabulary of units, prefixes, and relationships used by the library. The database is designed to be both human-readable and machine-parseable, enabling extensibility while preserving strict dimensional consistency.

    Unlike earlier versions of UDUNITS that used a custom text format, UDUNITS-2 stores unit definitions in XML, allowing modular composition of unit systems through standardized elements and import relationships.

    The XML files are loaded by the library at runtime to construct an internal representation of units, prefixes, and dimensions used for parsing unit expressions and performing conversions.

    3.1 XML Format

    Role of XML in Defining the Unit Database

    The UDUNITS-2 database uses XML as a structured representation of unit definitions, prefixes, and dimensional relationships. XML provides:

    • A hierarchical format suitable for representing relationships between units
    • A standardized syntax that can be validated for structural correctness
    • Extensibility through modular file composition
    • Explicit encoding of unit semantics

    The root element of the database is <unit-system>, which aggregates unit definitions and organizes them into a coherent dimensional framework.

    Example root file:

    <unit-system>
    <import>udunits2-prefixes.xml</import>
    <import>udunits2-base.xml</import>
    <import>udunits2-derived.xml</import>
    <import>udunits2-accepted.xml</import>
    <import>udunits2-common.xml</import>
</unit-system>

    This structure demonstrates that the primary XML file acts as an entry point that assembles multiple component files via <import> elements.

    Schema Structure and Key Elements

    The XML schema expresses unit definitions using structured elements that describe dimensional relationships, naming conventions, and conversion semantics.

    Key elements include:

    <unit>

    Defines an individual unit, including its dimensional basis and symbolic representation.

    Typical components:

    • <name> — canonical unit name
    • <symbol> — abbreviated representation
    • <definition> — semantic description of the unit
    • <base/> — indicates an SI base unit
    • <aliases> — alternate names for the unit

    Example:

    <unit>
    <base/>
    <name>
        <singular>meter</singular>
    </name>
    <symbol>m</symbol>
</unit>

    SI base units are defined in udunits2-base.xml.

    <dimension>

    Represents the dimensional composition of a unit in terms of base dimensions (e.g., length, mass, time).

    Dimensions allow the library to perform dimensional analysis to determine unit compatibility and construct conversion pathways.

    Examples of dimensional relationships include:

    • velocity = length / time
    • force = mass × length / time²

    <prefix>

    Defines scaling prefixes that modify unit magnitude.

    Examples include:

    • kilo (10³)
    • milli (10⁻³)
    • micro (10⁻⁶)

    Prefixes are defined separately from units and applied generically across compatible base units.

    Prefix definitions reside in udunits2-prefixes.xml.

    <import>

    Specifies inclusion of additional XML files into the unit system.

    The <import> mechanism enables modular database construction and allows custom extensions without modifying core definitions.

    Example:

    <import>udunits2-derived.xml</import>

    The root file imports multiple XML components that collectively define the full unit vocabulary.

    <alias>

    Defines alternate names or symbols for a unit, allowing multiple representations of equivalent concepts.

    Aliases improve interoperability with domain-specific conventions and legacy naming patterns.

    Example:

    <aliases>
    <name>
        <singular>metre</singular>
    </name>
</aliases>

    Aliases allow UDUNITS to recognize variant spellings and symbols while maintaining a single canonical definition.

    Validation Expectations

    XML files must conform to the structural expectations of the UDUNITS parser:

    • Elements must be properly nested and well-formed XML
    • Units must define consistent dimensional relationships
    • Imported files must exist and be accessible at runtime
    • Definitions must not introduce conflicting dimensional mappings

    Errors in XML structure or dimensional consistency typically result in failures when initializing a ut_system object via the library.

    Because the XML database is loaded dynamically, validation errors generally occur during initialization rather than compile time.

    3.2 Modular Organization

    Separation of Definitions Across Multiple XML Files

    The UDUNITS database is intentionally divided into multiple XML files to improve maintainability and enable selective extension of the unit vocabulary.

    Core definitions are separated into logical groupings:

    File Purpose
       
    udunits2-prefixes.xml Defines scale prefixes
    udunits2-base.xml SI base units
    udunits2-derived.xml Derived SI units
    udunits2-accepted.xml Units accepted for use with SI
    udunits2-common.xml Common non-SI units
    udunits2.xml Root aggregation file

    The root file imports component definitions using <import> elements.

    Import Relationships Between Files

    The XML database follows a hierarchical import structure:

    udunits2.xml
│
├── udunits2-prefixes.xml
├── udunits2-base.xml
├── udunits2-derived.xml
├── udunits2-accepted.xml
└── udunits2-common.xml

    Each imported file contributes additional unit definitions that are merged into a single in-memory unit system.

    This modular approach allows:

    • domain-specific extensions
    • replacement or augmentation of definitions
    • separation of SI and non-SI units
    • independent maintenance of prefixes and units

    The import-based architecture also enables downstream projects to supply custom unit definitions without modifying the upstream distribution.

    Example Hierarchy

    Typical conceptual layering:

    Core unit definitions

    Fundamental SI units:

    • meter
    • kilogram
    • second
    • kelvin
    • mole
    • ampere
    • candela

    Defined in:

    udunits2-base.xml

    Derived units

    Units constructed from base dimensions:

    • newton
    • joule
    • pascal
    • watt

    Defined in:

    udunits2-derived.xml

    Domain-specific extensions

    Units commonly used in specific scientific domains:

    • astronomical units
    • climatology units
    • geophysical units

    Defined in:

    udunits2-common.xml

    3.3 Default Database Location

    Typical Installed Locations

    After installation, the XML database is typically installed within the shared data directory:

    share/udunits/udunits2.xml

    Applications locate this file automatically or through the environment variable:

    UDUNITS2_XML_PATH

    If the XML database is installed in a non-standard location, the environment variable may be required to allow the library to locate the unit definitions.

    Source Repository Locations

    Within the UDUNITS-2 source tree, XML database files are located in the lib/ directory:

    lib/udunits2.xml
lib/udunits2-base.xml
lib/udunits2-derived.xml
lib/udunits2-prefixes.xml
lib/udunits2-common.xml

    The root XML file references component files via relative paths using <import> elements.

    Because the XML files are part of the source distribution, they can be inspected or modified prior to installation to customize supported units.
