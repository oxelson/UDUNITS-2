    Package Manager Installation

    UDUNITS-2 Documentation Package Manager Installation

    This page describes installing UDUNITS-2 using distribution-provided system package managers on Unix-like platforms.

    Note: Package-manager installations are provided and maintained by Linux distributions and macOS package ecosystems. They are not produced or versioned directly by the UDUNITS-2 upstream project.

    When Should You Use a Package Manager?

    Using a system package manager is usually the simplest and most reliable way to install UDUNITS-2. Consider this approach when:

    • You want the easiest installation method with minimal configuration.
    • You prefer the package manager to handle dependencies automatically.
    • You want UDUNITS-2 installed in standard system locations.
    • You want updates and security patches managed by the operating system.
    • You are installing UDUNITS-2 on a developer workstation or server where the distribution’s maintained packages are sufficient.

    A package manager installation is typically the best choice unless you need custom build options, a nonstandard installation prefix, or tighter control over the build environment.

    Supported Platforms

    UDUNITS-2 is widely available on 64-bit Unix-like architectures such as x86_64 and aarch64 with a POSIX environment.

    Platform Status Notes
    Linux (most distributions) ✅ Supported Available via distro package managers
    macOS ✅ Supported Via Homebrew or MacPorts
    Windows ⚠️ Limited No native winget package
    Note: There is no native Windows package-manager support for UDUNITS-2 via winget. For most Windows users, building from source or using Conda-based workflows are the simplest approach.

    Runtime Vs Development Packages

    Most distributions split UDUNITS-2 into two categories of packages:

    • Runtime packages – required to run the udunits2 utility or applications that already link to it
    • Development packages – required to build software that links against UDUNITS-2

    When Do You Need Development Packages?

    You need development packages if you are:

    • Compiling C or C++ applications
    • Building Python, R, or other language bindings from source
    • Linking UDUNITS-2 into another library (e.g., NetCDF builds)

    If you only run software that already depends on UDUNITS-2, runtime packages are sufficient.

    Dependencies

    Core Runtime Dependencies

    UDUNITS-2 depends on:

    Component Purpose
    Expat XML parser Parses the units database (udunits2.xml)
    Standard C library System-provided (glibc on Linux, libc on macOS)
    Note: Expat is almost always installed automatically as a dependency when installing UDUNITS-2.

    Compilers (Development Use Only)

    To build software against UDUNITS-2, you need a C compiler:

    • gcc or clang
    • Must support ISO C99 or later

    For C++ projects, compile your application with a C++ compiler (g++, clang++) while linking against the C-based UDUNITS-2 library.

    UDUNITS-2 Package Names By Distribution

    Package names and layouts vary by distribution. The table below lists common and typical names, but exact availability depends on distro version and enabled repositories.

    Distribution Runtime Package(s) Development Package(s) Notes
    RHEL / Rocky / Alma / CentOS udunits2 udunits2-devel May require EPEL on older releases
    Fedora udunits2 udunits2-devel Standard Fedora repos
    Debian / Ubuntu udunits-bin, libudunits2-0 libudunits2-dev Packages may be in universe
    SUSE / openSUSE udunits2 udunits2-devel RPM-based naming
    Arch Linux udunits (included) Headers included in main package
    macOS (Homebrew) udunits (included) Runtime and headers together
    macOS (MacPorts) udunits2 (included) Unified package
    Tip: If a package name differs, use your package manager to search. E.g.: apt search udunits or dnf search udunits

    Example Installation Commands

    Platform Just Run udunits2 Build Against UDUNITS-2
    Ubuntu / Debian sudo apt install udunits-bin sudo apt install libudunits2-dev
    Fedora sudo dnf install udunits2 sudo dnf install udunits2-devel
    RHEL / Rocky / Alma sudo dnf install udunits2 sudo dnf install udunits2-devel
    SUSE / openSUSE sudo zypper install udunits2 sudo zypper install udunits2-devel
    Arch Linux sudo pacman -S udunits (included)
    macOS (Homebrew) brew install udunits (included)
    macOS (MacPorts) sudo port install udunits2 (included)

    Files Installed

    A typical package manager installation of UDUNITS-2 provides:

    File Purpose
    udunits2 Command-line utility
    libudunits2 Shared library (.so / .dylib)
    udunits2.h C header (development packages only)
    udunits2.xml Units database

    Locations follow distribution filesystem policies and may differ across systems.

    Environment Variables

    When installed via a package manager, manual environment configuration is usually unnecessary.

    Variable Typical Use
    LD_LIBRARY_PATH Rarely needed; avoid overriding system paths
    UDUNITS2_XML_PATH Only set when using a custom XML database
    Warning: Overriding these variables unnecessarily is a common cause of runtime errors.

    Verifying Installation

    After installing UDUNITS-2, verify that the installation is working before building dependent software.

    A correct installation requires the following components to be discoverable:

    • the udunits2 executable
    • the development headers (udunits2.h)
    • the UDUNITS-2 library (libudunits2)
    • the units database (udunits2.xml)

    The Installation Health Check guide provides commands and scripts that confirm these components are installed and accessible. It can quickly detect common issues such as missing headers, missing libraries, incorrect include or library paths, or a misconfigured units database.

    Troubleshooting

    Most installation issues when installing UDUNITS-2 using system package managers fall into a few common categories.

    Examples of package managers include apt, dnf/yum, Homebrew, and MacPorts.

    Before using the troubleshooting tables below, run the checks in the UDUNITS-2 Installation Health Check to verify that the UDUNITS-2 executable, headers, libraries, and units database are correctly installed.

    Development Headers Missing

    Symptom Cause Diagnosis Fix
    udunits2.h: No such file or directory Development headers not installed Check installed packages Install development package (libudunits2-dev, udunits2-devel, etc.)
    Python/R packages fail detecting UDUNITS-2 Runtime package installed but headers missing Inspect build logs for udunits2.h errors Install the UDUNITS-2 development package

    Package Metadata Issues

    Symptom Cause Diagnosis Fix
    Package 'udunits2' not found from pkg-config pkg-config metadata missing Run pkg-config --list-all | grep udunits Install development package containing udunits2.pc

    Library Linking Issues

    Symptom Cause Diagnosis Fix
    ld: cannot find -ludunits2 Linker cannot find the UDUNITS-2 library Run pkg-config --libs udunits2 Install development package or add correct -L path
    undefined reference to ut_* functions Program not linking against UDUNITS-2 Inspect compile command Add -ludunits2 to linker flags

    Runtime Issues

    Symptom Cause Diagnosis Fix
    Runtime error loading libudunits2.so / .dylib Dynamic linker cannot find library Run ldd program or otool -L program Update linker cache (ldconfig) or adjust library paths
    Couldn't open units database XML database path overridden or missing Check echo $UDUNITS2_XML_PATH Unset variable or reinstall package

    Environment Conflicts

    Symptom Cause Diagnosis Fix
    Software builds but behaves unexpectedly Multiple UDUNITS-2 installations present Run which udunits2 and inspect linked libraries Remove duplicate installs or ensure consistent include/library paths

    Best Practices

    • Use one installation method (package manager or source build)
    • Avoid manual file deletion in system paths
    • Verify installation immediately after install is complete
    • Keep packages updated using your package manager

    NetCDF + UDUNITS-2 Integration

    Most Linux and macOS package managers install UDUNITS-2 automatically as a dependency of NetCDF, so installing NetCDF typically provides the required libraries along with the udunits2.xml units database.

    If you plan to build software against NetCDF, you should also install the UDUNITS-2 development headers (for example, libudunits2-dev on Debian/Ubuntu or udunits2-devel on Fedora/RHEL).

    You can verify that NetCDF was built with UDUNITS-2 support by running nc-config --has-udunits (which should return yes) or by inspecting a file with ncdump -h some_file.nc.

    As a general rule, install NetCDF and UDUNITS-2 from the same package repository and avoid mixing system packages with manually compiled libraries.

    Language Bindings Notes

    UDUNITS-2 issues are most often encountered indirectly, when installing language bindings or higher-level libraries—most commonly R, Python, and NetCDF-based software. In most cases, failures are rarely caused by UDUNITS-2 itself and are typically due to missing development packages.

    When using system Python or system R, ensure that all required development headers are installed before installing language bindings or building dependent software.

    Missing development headers for UDUNITS-2, Expat, or NetCDF account for a large fraction of reported language-binding installation failures.
